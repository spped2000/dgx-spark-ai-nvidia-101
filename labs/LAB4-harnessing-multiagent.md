## 0. ภาพรวมก่อนเริ่ม (อ่านให้ผู้เรียนฟัง 2 นาที)

ใน LAB 1–3 เราเห็นว่า DGX Spark คือ "สมอง" (Model) ที่ตอบคำถามได้ และเราจูนให้ถูกเงิน (Optimize Tokens) มาแล้ว

แต่แชตบอตเฉย ๆ มันแค่ "พูด" — มันยัง **ลงมือทำ** ไม่ได้ เช่น เปิดไฟล์ รันคำสั่ง ค้น GitHub หรือส่ง Slack

**LAB 4 คือบทที่เราเปลี่ยน "โมเดลที่พูดได้" ให้กลายเป็น "agent ที่ลงมือทำได้"**

### Key message (TH)
> **Agent = Model + Harness.** โมเดลอย่างเดียวแค่ทำนายคำถัดไป แต่ "harness" คือเปลือกที่ห่อโมเดลไว้ ให้มันวนลูป **คิด → ทำ → ดูผล → คิดต่อ** จนงานเสร็จ — นี่คือสิ่งที่ทำให้ AI "ทำงานเป็น" ไม่ใช่แค่ "ตอบเป็น"

### Analogy (TH)
> โมเดล = **พ่อครัวเก่ง ๆ ที่ถูกมัดมือไว้** บอกสูตรได้ทุกอย่างแต่หยิบจับอะไรไม่ได้
> Harness = **ครัว + มือ + ตา** ให้พ่อครัวหยิบมีด (tool), ชิมรส (observe), แล้วปรุงต่อ (reason → act) จนได้จานเสร็จ
> Multi-agent = **ครัวร้านใหญ่** มี **หัวหน้าเชฟ (orchestrator)** สั่งงาน **เชฟทำอาหาร (coding agent)** กับ **คนชิม/ตรวจจาน (reviewer agent)** คนละหน้าที่ แล้วส่งงานต่อกัน

## 1. แกนทฤษฎี: "Agent = Model + Harness" (instructor ฉายสไลด์ 5 นาที)

เปิดสไลด์แกนกลางของทั้งคอร์ส แล้วชี้ว่า **LAB นี้ = ตัว Harness โดยตรง** (Pillar 2)

```
                    ┌─────────── HARNESS ───────────┐
                    │                                │
   Prompt ──┐       │     CONTEXT ──► OBSERVE        │
            │       │        ▲           │           │
  Memory ───┼──────►│        │           ▼           │◄── Tools & Skills
            │       │      ACT  ◄───── REASON         │
 Orchestr. ─┘       │                                │
                    └────────────────────────────────┘
                       ห่อด้วย: Security & Governance
```

**อธิบายลูป CONTEXT → OBSERVE → REASON → ACT แบบบ้าน ๆ:**
- **CONTEXT** — agent ได้โจทย์ + ความจำ + รายการ tool ที่ใช้ได้
- **OBSERVE** — มันดูสถานการณ์ตอนนี้ (ผลจากรอบที่แล้ว / ไฟล์ / error)
- **REASON** — มันคิดว่า "ขั้นต่อไปควรทำอะไร" (เลือก tool ไหน ใส่ค่าอะไร)
- **ACT** — มันเรียก tool จริง แล้วผลที่ได้กลับเข้า OBSERVE → วนใหม่จนเสร็จ

**จุดที่ต้องย้ำให้มือใหม่จำ:**
> **single-agent** = หนึ่งโมเดลวนลูปนี้คนเดียว + มี tool หลายตัว
> **multi-agent** = มีหลาย agent แต่ละตัววนลูปนี้ของตัวเอง โดยมี **orchestrator** เป็นคนแจกงานและรวมผล

แมปกลับ 3 เสา:
- **Pillar 1 (Optimize Tokens)** = ฝั่ง cost/latency ของ "Model" — ลูปยิ่งยาว ยิ่งกิน token ยิ่งแพง
- **Pillar 2 (Harnessing)** = ตัว Harness นี้แหละ ← **เรากำลังอยู่ตรงนี้**
- **Pillar 3 (Secure & Governance)** = กล่องที่ห่อ harness ไว้ (LAB 5)

## 3. ส่วนที่ผู้เรียนลงมือเอง — Single-Agent Tool-Use (MCP) (15 นาที, no-code)

**เป้าหมาย:** ให้ผู้เรียนทุกคนเห็น "โมเดลที่พูดได้" กลายเป็น "agent ที่หยิบ tool มาทำงานได้" — ด้วยการคลิก Run-Cell ล้วน ไม่ต้องเขียนโค้ด

> โน้ตบุ๊กถูกเตรียมไว้ล่วงหน้าใน JupyterHub ของแต่ละคนแล้ว (ชื่อ `LAB4_single_agent.ipynb`) — ผู้เรียนแค่ "เปิด → กด Run ทีละเซลล์ → อ่านผล"

### ขั้นที่ 3.1 — เปิดโน้ตบุ๊กแล้วดู agent ตอนยังไม่มี tool (baseline)
รันเซลล์แรก (instructor นับ 3-2-1 พร้อมกันทั้งห้องไม่ได้ตรงนี้ — ดูเรื่องเวฟในข้อ 5) ถามคำถามที่โมเดลตอบเองไม่ได้ เช่น:

> "ตอนนี้กี่โมง และในโฟลเดอร์งานมีไฟล์อะไรบ้าง?"

**ผลที่คาดหวัง:** โมเดลตอบทำนองว่า "ผมไม่รู้เวลาจริง และเปิดไฟล์ไม่ได้" หรือ **เดามั่ว** → ทำให้ผู้เรียนเห็นปัญหาด้วยตาตัวเอง ว่าโมเดลเปล่า ๆ "ตาบอด มือเปล่า"

### ขั้นที่ 3.2 — เปิด tool ให้ agent ผ่าน MCP แล้วถามใหม่
รันเซลล์ถัดไปที่ "เสียบ tool" เข้ากับ agent ผ่าน **MCP (Model Context Protocol)** — มาตรฐานเดียวกับใน AI-Q Blueprint ข้อ 2.1

tool ที่เปิดให้ (อ่าน-อย่างเดียว ปลอดภัย):
- `get_time` — ขอเวลาปัจจุบัน
- `list_files` — ดูรายชื่อไฟล์ในโฟลเดอร์งาน
- `read_file` — อ่านเนื้อไฟล์

ถามคำถามเดิมซ้ำ แล้ว **ดูที่ panel "agent steps"** ในโน้ตบุ๊ก — มันจะโชว์ลูปทีละก้าวให้เห็น:

```
[step 1] REASON: ผู้ใช้ถามเวลา → ต้องเรียก get_time
[step 1] ACT   : get_time() ──► "2026-06-20 14:02"
[step 2] REASON: ผู้ใช้ถามไฟล์ → ต้องเรียก list_files
[step 2] ACT   : list_files() ──► ["note.txt", "data.csv"]
[step 3] REASON: ข้อมูลครบแล้ว → ตอบผู้ใช้
[step 3] DONE  : "ตอนนี้ 14:02 น. ในโฟลเดอร์มี note.txt และ data.csv ครับ"
```

**ผลที่คาดหวัง:** ตอบถูกเป๊ะ และผู้เรียน **เห็นลูป CONTEXT → OBSERVE → REASON → ACT ทำงานจริง** ตรงกับสไลด์ข้อ 1

> **ประโยคที่ใช้พูด:** "โมเดลตัวเดิมเป๊ะ ๆ ไม่ได้ฉลาดขึ้นเลย — แต่พอเราใส่ harness (tool + ลูป) เข้าไป มันทำงานได้ทันที นี่คือความหมายของ Agent = Model + Harness"

### ขั้นที่ 3.3 — ลองโจทย์ที่ต้องวนหลายก้าว
รันเซลล์สุดท้ายของส่วนนี้ ถามโจทย์ที่ต้องใช้หลาย tool ต่อกัน เช่น:

> "เปิด note.txt แล้วสรุปให้หน่อยว่าเขียนเรื่องอะไร"

agent จะวน: `list_files` → `read_file("note.txt")` → สรุป → ตอบ

**ผลที่คาดหวัง:** เห็น agent วนลูป **2–3 ก้าว** (จำกัดไว้ตามข้อ 5) แล้วจบงาน

> ⚙️ **พารามิเตอร์ที่ตั้งไว้ในโน้ตบุ๊ก (กัน contention):** `max_steps = 3`, `max_tokens = 128–256` ต่อก้าว — ผู้เรียนไม่ต้องแก้ แค่รู้ว่าเราจงใจ "ตีกรอบลูป" ไม่ให้มันคิดวนยาวจนกิน GPU ทั้งห้อง

## 4. ส่วนที่ instructor โชว์ — Full Multi-Agent Loop (10 นาที, ดูอย่างเดียว)

ส่วนนี้ **instructor รันให้ดูครั้งเดียวบนจอใหญ่** ผู้เรียน **ไม่ต้องรันเอง** (contention สูงสุดของทั้งคอร์ส — ดูเหตุผลข้อ 5)

### โครงที่จะโชว์: Orchestrator → Coding Agent + Reviewer Agent
ใช้ตัวอย่าง production จาก NemoClaw Hermes (ข้อ 2.2) แต่ย่อลงมาให้รันบน Spark เครื่องเดียว:

```
                         ┌─────────────────────┐
                         │   ORCHESTRATOR      │  (สีน้ำเงิน)
                         │  "ฉันจะเขียนฟังก์ชัน │
                         │   บวกเลข พร้อมตรวจ"   │
                         └──────────┬──────────┘
                    แจกงาน          │          รวมผล
                  ┌─────────────────┴─────────────────┐
                  ▼                                   ▼
       ┌────────────────────┐            ┌────────────────────┐
       │   CODING AGENT     │            │   REVIEWER AGENT    │
       │   (สีเขียว)         │  ───code──►│   (สีส้ม)            │
       │  เขียนฟังก์ชัน add() │            │  ตรวจ + ติชม + อนุมัติ │
       └────────────────────┘            └────────────────────┘
```

**สิ่งที่ผู้เรียนจะเห็นบนจอ (ข้อความแต่ละ agent โชว์คนละสี):**

| สี | agent | ทำอะไร (ตัวอย่าง) |
|---|---|---|
| 🔵 น้ำเงิน | **Orchestrator** | "งานคือเขียนฟังก์ชันบวกเลข ฉันจะให้ coding agent เขียน แล้วให้ reviewer ตรวจ" |
| 🟢 เขียว | **Coding Agent** | "เขียนแล้ว: `def add(a, b): return a + b`" |
| 🟠 ส้ม | **Reviewer Agent** | "โอเค ถูกต้อง แต่ควรเพิ่ม type hint → `def add(a: int, b: int) -> int:`" |
| 🔵 น้ำเงิน | **Orchestrator** | "รับ feedback แล้ว ส่งแก้ → อนุมัติ → จบงาน" |

**ผลที่คาดหวัง:** ผู้เรียนเห็น **delegation (การแจกงาน) คนละสีชัด ๆ** และเข้าใจว่า multi-agent ไม่ใช่เวทมนตร์ — มันคือ "หลาย single-agent คุยกันผ่าน orchestrator"

> **ประโยคที่ใช้พูด:** "สังเกตว่า reviewer ไม่ใช่คนเดียวกับคนเขียน — เหมือนเรามี QA แยกจาก dev นี่คือเหตุผลที่ multi-agent ให้งานคุณภาพดีกว่า single-agent: มันมี 'คนตรวจการบ้าน' ในตัว"

### โมเดลที่ instructor ใช้ในส่วนนี้
- **โมเดลเล็กผ่าน Ollama** (ไม่ใช่โมเดล vLLM ที่ทั้งห้องแชร์) — เพราะ multi-agent ยิงหลายลูปต่อเนื่อง ถ้าใช้ endpoint รวมจะแย่ง bandwidth ทั้งห้อง
- จำกัด **2–3 steps** ต่อ agent, `max_tokens 128–256` — ให้จบเร็ว เห็นภาพครบ ไม่ค้าง

> ⚠️ **ห้ามโหลดโมเดลที่สองพร้อมกันบน vLLM** — Ollama ตัวเล็กรันแยกได้ แต่ระวังอย่าให้ชนกับโมเดลหลักจนเบียด memory bandwidth (คอขวดจริงของ GB10) ดู fallback ข้อ 6

## 5. ทำไมต้องระวัง "contention" และแบ่งเวฟ (instructor อธิบาย 2 นาที + คุมหน้างาน)

LAB 4 คือ **lab ที่กิน GPU หนักที่สุด** เพราะ:
- agent วนหลายลูป = ยิงโมเดลหลายครั้งต่อหนึ่งโจทย์ (ไม่ใช่ถาม-ตอบครั้งเดียวเหมือน LAB 1)
- ถ้า 30 คนยิง multi-agent พร้อมกัน = แย่ง memory bandwidth กันจน **ช้าหมดทั้งห้อง / เสี่ยง OOM**

### กลยุทธ์คุม contention (ทำตามนี้)
1. **ผู้เรียนรันแค่ single-agent (ข้อ 3)** ซึ่งเบากว่ามาก — full multi-agent ให้ instructor โชว์อย่างเดียว (ข้อ 4)
2. **แบ่งเวฟ** สำหรับข้อ 3: แบ่งห้องเป็น **2 เวฟ (15 คน × 2)** — เวฟ A กด Run ก่อน, เวฟ B รออีก ~1 นาทีค่อยกด → ดูได้จาก dashboard ว่า batch ไม่พีคพร้อมกัน
3. **จำกัดลูป**: `max_steps = 2–3`, `max_tokens = 128–256` — ตั้งไว้ในโน้ตบุ๊กแล้ว ผู้เรียนไม่ต้องแตะ
4. **โมเดลเล็ก Ollama** สำหรับ multi-agent loop ของ instructor — แยกจาก endpoint รวม
5. **ฉาย dashboard ค้างไว้** (Triton → Prometheus → Grafana) ให้เห็น batch รวมตอนเวฟยิง

> **ประโยคที่ใช้พูด (สคริปต์มาตรฐานทั้งคอร์ส):** "ถ้ามันช้าลงตอนหลายคนยิงพร้อมกัน — นั่นคือ **บทเรียน Pillar 1 ไม่ใช่บั๊ก** agent ที่วนลูปเยอะกิน token เยอะ นี่คือเหตุผลว่าทำไม cost per token ถึงเป็น KPI"

## 6. Fallback (ถ้าหน้างานมีปัญหา)

| อาการ | สาเหตุที่น่าจะเป็น | ทำอะไร |
|---|---|---|
| ข้อ 3 ช้ามากตอนเวฟยิง | batch รวมพีค / bandwidth เต็ม | ลดเหลือเวฟละ 10 คน (3 เวฟ), ย้ำว่า "ช้า = บทเรียน" |

| OOM ตอน instructor รัน multi-agent | Ollama + vLLM เบียด memory | **ห้ามโหลดโมเดลที่สอง** — ใช้คลิปสำรองที่อัดไว้แทนการรันสด |
| Ollama โหลดช้า/ล่ม | โมเดลยังไม่ warm | pre-pull + re-warm ก่อนเริ่ม block (ดู runbook); ถ้ายังไม่ได้ → ใช้ **cloud fallback** (endpoint สำรองบนคลาวด์) เฉพาะส่วนโชว์ของ instructor |
| เน็ตล่ม / endpoint รวมล่ม | LAN/serving มีปัญหา | single-agent ข้อ 3 รันบนโมเดลโลคัลได้; ถ้าจอ instructor ก็ล่ม → เปิด **คลิปสำรอง** ของ multi-agent loop |
| เวลาไม่พอ (ช้ากว่าแผน) | บล็อกก่อนหน้าลาก | **ตัด full multi-loop ข้อ 4 เหลือ instructor เล่าจากภาพนิ่ง/คลิป** แล้วเก็บข้อ 3 (ผู้เรียนลงมือ) ไว้เสมอ — ข้อ 3 คือ visible win ที่ต้องได้ |

## 7. สรุป LAB 4 (instructor ปิดท้าย 2 นาที)

ให้ผู้เรียนพูดตามได้ 3 ประโยค:
1. **Agent = Model + Harness** — โมเดลคือสมอง, harness คือมือ+ตา+ลูป
2. **Harness คือลูป CONTEXT → OBSERVE → REASON → ACT** ที่หยิบ tool มาทำงานจริง (เราเห็นเองในข้อ 3 ผ่าน MCP)
3. **Multi-agent = orchestrator แจกงานให้ coding agent + reviewer agent** (เหมือน dev + QA) — ของจริงคือ NemoClaw Hermes: orchestrator=Ultra, sub-agents=Nano
