## 0. ปูพื้นก่อนเริ่ม — "สามชื่อที่ฟังคล้ายกันแต่คนละหน้าที่"

ก่อนจะลงมือ ขออธิบายปัญหาที่มือใหม่ทุกคนสับสน: NVIDIA มีของสามอย่างที่ชื่อขึ้นต้นด้วย "Nemo/NeMo" คล้ายกันมาก แต่หน้าที่คนละเรื่องเลย ถ้าแยกสามอันนี้ออกได้ คุณจะเข้าใจภาพรวมทั้งคอร์สทันที

### Analogy: ครัวร้านอาหาร

ลองนึกว่าเรากำลังเปิดร้านอาหาร:

- **NeMo** = **ครัว + เตา + สูตรอาหาร** ที่เอาไว้ "ทำอาหารจานใหม่" (เอาไว้ฝึก/ปรับโมเดล) — ลูกค้าไม่เคยเห็นครัว
- **Nemotron** = **จานอาหารที่เสิร์ฟจริง** ที่ลูกค้ากิน(โมเดลที่รันตอบคำถามจริง ๆ) — อันนี้คือสิ่งที่เราจะเล่นในแล็บนี้
- **NeMo Guardrails** = **ยามหน้าร้าน + กฎห้ามนำของอันตรายเข้าร้าน** (ด่านความปลอดภัยกรองคำถาม/คำตอบ) — เดี๋ยวลงรายละเอียดใน LAB 5 ตอนบ่าย

| ชื่อ (English) | คืออะไร (ภาษาไทย) | ตอนไหนที่ใช้ | เราเล่นใน LAB ไหน |
|---|---|---|---|
| **NeMo** | กล่องเครื่องมือ "สร้าง/ปรับโมเดล" (train, fine-tune, export) | ตอนพัฒนาโมเดล (ก่อนเอามาใช้) | concept-only (เกริ่นใน LAB นี้) |
| **Nemotron** | ตัวโมเดลจริงที่รันตอบคำถาม (LLM family ของ NVIDIA) | ตอนใช้งาน (serving / inference) | **LAB 2 (อันนี้)** |
| **NeMo Guardrails** | ด่านความปลอดภัยกรอง input/output ของโมเดล | ตอนใช้งาน (ป้องกัน jailbreak) | LAB 5 (ช่วงบ่าย) |

> **Key message (TH):** "NeMo = ทำโมเดล · Nemotron = ใช้โมเดล · NeMo Guardrails = คุมโมเดล" — สามอันนี้ทำงานคนละช่วงเวลา ไม่ใช่คู่แข่งกัน

## 1. Objective — จบ LAB นี้แล้วผู้เรียนจะ...

1. **แยกออก** ว่า **NeMo** (สร้างโมเดล) ต่างจาก **Nemotron** (โมเดลที่รัน) ต่างจาก **NeMo Guardrails** (ด่านความปลอดภัย) อย่างไร
2. **เข้าใจ MoE (Mixture-of-Experts)** แบบง่าย ๆ — ทำไม **Nano 30B-A3B** ตัวใหญ่ 30B แต่ "เร็วเหมือนตัวเล็ก 3B" และทำไมจึง **ฟิตกับ DGX Spark 128GB**
3. **เห็นความต่างด้วยตาตัวเอง** ระหว่าง **`/think`** (โมเดลคิดก่อนตอบ) กับ **`/no_think`** (ตอบทันที) โดยส่งคำถามเดียวกัน — **คลิกล้วน ไม่ต้องเขียนโค้ด**
4. **ตัดสินใจเป็น** ว่าเมื่อไรควรเปิด reasoning เมื่อไรควรปิด (trade-off ความฉลาด vs ความเร็ว/ต้นทุน token)

> **Key message (TH):** "โมเดลตัวเดียวกัน เปิด/ปิดสมองได้ — เปิดสมองตอบฉลาดขึ้นแต่ช้าลงและกิน token มากขึ้น เลือกใช้ให้ถูกงาน"

## 2. อธิบาย MoE / Active Params แบบง่ายที่สุด

ชื่อรุ่นเขียนว่า **Nemotron Nano 30B-A3B** — ตัวเลขสองตัวนี้สำคัญมาก:

- **30B** = จำนวน "พารามิเตอร์ทั้งหมด" (total parameters) ที่โมเดลมี = ขนาดสมองทั้งก้อน
- **A3B** = **A**ctive **3B** = จำนวนพารามิเตอร์ที่ "ถูกปลุกมาทำงานจริงต่อ 1 คำตอบ" (active parameters)

### MoE คืออะไร — Analogy: ทีมผู้เชี่ยวชาญ

ลองนึกว่าโมเดลคือ **บริษัทที่มีผู้เชี่ยวชาญ (experts) อยู่ 30 คน** แต่ละคนเก่งคนละเรื่อง (คนนี้เก่งคณิต คนนั้นเก่งภาษา คนโน้นเก่งโค้ด)

- โมเดลแบบเก่า (**Dense**) = ทุกคำถามต้องเรียก **ผู้เชี่ยวชาญทั้ง 30 คน** มาประชุมพร้อมกัน → ฉลาด แต่ช้าและเปลืองมาก
- โมเดลแบบ **MoE (Mixture-of-Experts)** = มี **"ผู้จัดการ" (router)** คอยดูคำถาม แล้วเรียกเฉพาะ **ผู้เชี่ยวชาญที่เกี่ยวข้อง ~3 คน** มาตอบ → **ฉลาดเท่าทีมใหญ่ แต่เร็วเหมือนทีมเล็ก**

นี่คือเหตุผลที่ **Nano 30B-A3B "เร็วเหมือน 3B แต่ฉลาดกว่า 3B"**:
- **ความเร็ว/ต้นทุนการคำนวณ** ขึ้นกับ **active params (3B)** → เร็วเหมือนโมเดลเล็ก
- **ความฉลาด/ความรู้** ขึ้นกับ **total params (30B)** → มีคลังความรู้ของตัวใหญ่

### ทำไมถึงฟิตกับ DGX Spark

- DGX Spark มี **memory 128GB** และคอขวดจริงคือ **memory bandwidth (~273 GB/s)** ไม่ใช่ความจุ
- โมเดล MoE active น้อย → คำนวณต่อ token น้อยลง → เคลื่อนข้อมูลผ่าน bandwidth น้อยลง → **เร็วบนเครื่องเดียวกัน**
- น้ำหนักโมเดล (FP8/NVFP4) ไม่กินเมมหมด เหลือที่ส่วนใหญ่ให้ **KV cache** (ความจำชั่วคราวระหว่างคุย) → รองรับหลายคนพร้อมกันได้

### Nemotron 3 family (ตามที่ประกาศในงาน GTC)

| รุ่น (English) | total / active | บทบาท (ภาษาไทย) | ใช้บน Spark ได้ไหม |
|---|---|---|---|
| **Nano 30B-A3B** | 30B / 3B | งานเร็ว ประหยัด, เหมาะเป็น sub-agent + มือใหม่ | ✅ เหมาะที่สุด (ตัวที่เราใช้) |
| **Super 120B-A12B** | 120B / 12B | throughput สูง + tool calling เก่ง | ◐ หนักแต่พอได้ |
| **Ultra 550B-A55B** | 550B / 55B | token efficiency สูงสุด (orchestrator) | ✗ เกินเครื่องเดียว — "ตามที่ประกาศในงาน GTC" |

> **A = active params** เสมอ ในชื่อรุ่นของตระกูลนี้
>
> Nemotron ยังมี **variants** อื่น ๆ ที่ปรับเฉพาะงาน: **Nano Omni** (multimodal), **RAG/Retriever**, **Speech**, **Safety** — หาดูได้ที่ `build.nvidia.com` หรือ `huggingface.co/nvidia`

> **Key message (TH):** "30B-A3B = สมอง 30B แต่จ่ายค่าไฟแค่ 3B" — นี่คือเทคนิคที่ทำให้โมเดลฉลาดรันไหวบนเครื่องเล็ก

## 3. Setup ฝั่งผู้เรียน (ก่อนเริ่มจับเวลา)

> ผู้เรียน **ไม่ต้องติดตั้งอะไร** ทุกอย่าง pre-provision ไว้แล้ว แค่เปิดเบราว์เซอร์

1. เปิดเบราว์เซอร์ ไปที่ URL ที่อยู่บน **การ์ดของคุณ** (Open WebUI ชี้มาที่ shared endpoint บน DGX Spark)
2. ใส่ `user` / `pass` จากการ์ด แล้วกด **Login**
3. ดูมุมบน ตรวจว่า model ที่เลือกอยู่คือ **Nemotron Nano** (instructor ตั้ง default ให้แล้ว — **ห้ามสลับไปโหลดโมเดลตัวอื่น** เพราะจะทำให้เครื่อง OOM ทั้งห้อง)

> ⚠️ **กติกาทั้งห้อง:** ทุกคนใช้ **โมเดลตัวเดียวกันตัวเดียว** การ toggle `/think` / `/no_think` เป็นแค่การ "เปิด-ปิดสมอง" ของโมเดลเดิม **ไม่ใช่การโหลดโมเดลใหม่** → ปลอดภัยกับเครื่อง

## 4. ขั้นตอนการทำ LAB (no-code, แบ่ง 2 เวฟ กัน contention)

เนื่องจากทุกคนยิงโมเดลตัวเดียวกันบนเครื่องเดียว ถ้า 30 คนกด **`/think`** พร้อมกัน คำตอบแบบคิดยาวจะกิน token เยอะและแย่ง bandwidth → เราจึง **แบ่งเป็น 2 เวฟ**

> **Wave A = ผู้เรียนครึ่งซ้ายของห้อง · Wave B = ผู้เรียนครึ่งขวา** (instructor ชี้แบ่งหน้างาน) — ทำให้คำตอบไม่หน่วงพร้อมกันทั้งห้อง และทุกคนเห็นผลชัด

### Step 1 — ตั้ง baseline ด้วย `/no_think` (ตอบเร็ว)

ทุกคน (ทั้ง Wave A และ B) ทำพร้อมกัน เพราะ `/no_think` เบา ไม่หน่วง:

1. ในช่องแชต พิมพ์คำถามนี้ **โดยขึ้นต้นด้วย `/no_think`** (คัดลอกได้เลย):

   ```
   /no_think มีกบ 3 ตัวนั่งบนขอนไม้ ตัวหนึ่งตัดสินใจจะกระโดดลงน้ำ ถามว่าตอนนี้เหลือกบกี่ตัวบนขอนไม้?
   ```

2. กด **Send** แล้วสังเกต:
   - โมเดล **ตอบทันที** ไม่มีขั้นตอนคิด
   - จดเวลาคร่าว ๆ (กี่วินาทีกว่าจะเริ่มมีตัวอักษรขึ้น)
   - จดว่าคำตอบ "ตรง/ผิด" อย่างไร (คำถามนี้เป็นกับดัก — "ตัดสินใจจะกระโดด" ยังไม่ได้กระโดดจริง)

### Step 2 — เปิดสมองด้วย `/think` (คิดก่อนตอบ) — ทำเป็นเวฟ

> **Wave A กดก่อน · Wave B รอ instructor นับ แล้วค่อยกด** (กัน 30 คนคิดยาวพร้อมกัน)

1. พิมพ์ **คำถามเดิมเป๊ะ ๆ** แต่เปลี่ยนหัวเป็น `/think`:

   ```
   /think มีกบ 3 ตัวนั่งบนขอนไม้ ตัวหนึ่งตัดสินใจจะกระโดดลงน้ำ ถามว่าตอนนี้เหลือกบกี่ตัวบนขอนไม้?
   ```

2. กด **Send** แล้วสังเกตความต่างกับ Step 1:
   - มี **บล็อก "การคิด" (thinking / reasoning)** ขึ้นมาก่อน — โมเดลพิมพ์ขั้นตอนการให้เหตุผลออกมา (เช่น "ตัดสินใจจะกระโดด ≠ กระโดดแล้ว ดังนั้นยังเหลือ 3 ตัว")
   - คำตอบ **เริ่มช้ากว่า** และ **ยาวกว่า** (ใช้ token มากกว่า)
   - คำตอบมักจะ **ถูกต้องกว่า** บนคำถามที่ต้องใช้เหตุผล

### Step 3 — ลองคำถามคณิตให้ความต่างชัดขึ้น (เลือกทำถ้ามีเวลา)

ส่งคำถามนี้ทั้งสองโหมด เปรียบเทียบกัน:

```
/no_think ร้านขายปากกาด้ามละ 7 บาท ซื้อ 3 แถม 1 ถ้าจ่ายเงินไป 100 บาท ซื้อปากกาให้ได้มากที่สุดจะได้กี่ด้ามและเหลือเงินเท่าไร?
```
```
/think ร้านขายปากกาด้ามละ 7 บาท ซื้อ 3 แถม 1 ถ้าจ่ายเงินไป 100 บาท ซื้อปากกาให้ได้มากที่สุดจะได้กี่ด้ามและเหลือเงินเท่าไร?
```

- `/no_think` มักจะ "เดา" หรือข้ามเงื่อนไข "ซื้อ 3 แถม 1"
- `/think` จะ "แตกขั้นตอน" ออกมาให้เห็นว่าคำนวณยังไง → คำตอบน่าเชื่อถือกว่า

> ⚠️ **toggle เป็นแค่การพิมพ์ `/think` หรือ `/no_think` นำหน้าคำถาม** — โมเดลเดิม ไม่โหลดอะไรเพิ่ม นี่คือ "คลิก/พิมพ์ล้วน" ตามที่โจทย์ต้องการ

## 5. ผลที่คาดหวัง (Expected Outcome)

| สังเกต | `/no_think` | `/think` |
|---|---|---|
| มีขั้นตอนการคิดโชว์ไหม | ไม่มี — ตอบเลย | มี — เห็น reasoning ก่อนคำตอบ |
| เริ่มตอบเร็วแค่ไหน (TTFT) | เร็วกว่า | ช้ากว่า (ต้องคิดก่อน) |
| ความยาวคำตอบ / token ที่ใช้ | น้อย | มาก |
| ความถูกต้องบนคำถามมีกับดัก | พลาดง่าย | ถูกต้องกว่า |
| เหมาะกับงานแบบไหน | ถาม-ตอบเร็ว, ดึงข้อมูลตรง ๆ, แชตทั่วไป | โจทย์เหตุผล, คณิต, วางแผนหลายขั้น |

**สิ่งที่ผู้เรียนควร "อ๋อ" ได้ตอนจบ:**
1. โมเดลตัวเดียวกัน "เปิด/ปิดสมอง" ได้ ด้วยการพิมพ์คำสั่งนำหน้า
2. คิดมาก = ฉลาดขึ้น **แต่** ช้าลง + กิน token มากขึ้น = **แพงขึ้น** → โยงเข้า Pillar 1 (Optimize Tokens, "cost per token = KPI ใหม่")
3. การเลือก `/think` vs `/no_think` คือการตัดสินใจ trade-off จริงในงาน production

> **Analogy ปิดท้าย (TH):** "`/no_think` = ตอบจากความเคยชิน (ปากไว) · `/think` = หยุดคิดให้รอบคอบก่อนพูด (แต่ใช้เวลา)" — คนเก่งจริงเลือกใช้ทั้งสองโหมดให้ถูกสถานการณ์

## 6. Fallback (ถ้ามีปัญหาหน้างาน)

| ปัญหา | ทางแก้ |
|---|---|

- **ใช้เวลา:** **39 วัน**
- **ผลที่ดีขึ้น:** คะแนน **MalayMMLU** ขยับจาก **50 → 77** (เก่งภาษามาเลเซียขึ้นชัดเจน)
- **⚠️ Trade-off ที่ต้องรู้ (teachable point):** พอ fine-tune ให้เก่งด้านหนึ่ง ความสามารถอีกด้าน **ตกลง** — คะแนน **C-MMLU (จีน) ตกลง**

**บทเรียนสำคัญ (ภาษาไทย):**
- การ fine-tune **ไม่ใช่ของฟรี** — "ปรับให้เก่งเรื่องหนึ่ง อาจแลกกับการลืมอีกเรื่อง" (เหมือนคนซ้อมหนักด้านหนึ่งจนละเลยอีกด้าน)
- นี่คือเหตุผลที่ต้อง **วัดผลหลายด้าน** ก่อน-หลัง fine-tune เสมอ ไม่ใช่ดูแค่ตัวเลขที่เราอยากเห็น
- งานทั้งหมดนี้อยู่ฝั่ง **NeMo** (สร้าง/ปรับโมเดล) ซึ่งคนละช่วงกับสิ่งที่เราทำในแล็บนี้ (ใช้ **Nemotron** ที่รันอยู่แล้ว)

> **เชื่อมโยง:** แล็บนี้เราเป็น "ลูกค้าที่กินจานอาหาร (Nemotron)" — ส่วน YTL คือ "เชฟที่เข้าครัว (NeMo) ปรุงจานใหม่" ใน 39 วัน

## 8. สรุปสั้น (Recap)
