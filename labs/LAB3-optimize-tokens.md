# LAB 3 — Pillar 1: Optimize Tokens (no-code)

> **ช่วงเวลา:** 13:10–13:50 (40 นาที) · **ชนิด:** Lab ลงมือทำ (no-code) · **กลุ่มเป้าหมาย:** มือใหม่เริ่มจากศูนย์
> **เครื่อง:** DGX Spark 1 เครื่อง (GB10 Grace Blackwell, 128GB unified memory) แชร์ทั้งห้อง ~30 คน
> **โมเดลที่ใช้:** โมเดลเดียวที่แชร์ทั้งห้อง (Nemotron Nano คลาส MoE / 7–9B, FP8) เสิร์ฟผ่าน vLLM (หรือ NIM)

## Key Message (สิ่งที่อยากให้จำกลับบ้าน)

ในยุค Agent **"cost per token = KPI ใหม่"** — เราไม่ได้วัดแค่ "ตอบถูกไหม" แต่วัดว่า **"ตอบเร็วแค่ไหน และเปลืองเท่าไหร่ต่อ 1 token"** ด้วย DGX Spark เครื่องเดียวที่ต้องเลี้ยง 30 คนพร้อมกัน ทุก token ที่เราประหยัดได้ = ที่ว่างให้เพื่อนอีกคนใช้งานได้

**Pillar 1 ตอบคำถามเดียว:** _"ทำไมเช้าช้า แล้วเราหมุนคันโยกอะไรได้บ้างให้มันเร็วขึ้น?"_

---

## Analogy (อุปมาเปิดใจมือใหม่)

ลองนึกว่า DGX Spark คือ **ร้านกาแฟร้านเดียว มีบาริสต้าคนเดียว** แต่มีลูกค้า 30 คนต่อแถวพร้อมกัน

- **บาริสต้า = GPU** — ทำกาแฟได้เร็วมาก แต่มีคนเดียว
- **เครื่องชงกาแฟ + ท่อส่งน้ำ = memory bandwidth (~273 GB/s)** — นี่คือ **คอขวดจริง** ไม่ใช่ตัวบาริสต้า ยิ่งต้องวิ่งไปหยิบเมล็ดกาแฟ (อ่าน weights จาก memory) บ่อย ๆ ยิ่งช้า
- **สูตรกาแฟที่จำไว้แล้ว = prefix cache** — ถ้าออเดอร์ขึ้นต้นเหมือนเดิม ("ลาเต้ร้อน...") บาริสต้าไม่ต้องเริ่มคิดใหม่
- **โต๊ะเตรียมของ = KV cache** — ที่วางของระหว่างทำ ถ้าโต๊ะเต็ม (memory เต็ม) รับลูกค้าเพิ่มไม่ได้

> **ประโยคทองของ Pillar 1:** _"ตอนยิงพร้อมกันแล้วช้าลง = บทเรียน ไม่ใช่บั๊ก"_ — เพราะบาริสต้าคนเดียวกำลังแบ่งเวลาให้ทุกคน

## ทำไม "เช้าช้า"? (Why is it slow?)

มือใหม่มักคิดว่า "GPU แรง = ต้องเร็วเสมอ" แต่ความจริงคือ:

1. **คอขวด = memory bandwidth ไม่ใช่ความเร็วการคำนวณ** — ทุกครั้งที่โมเดลสร้าง 1 token ใหม่ มันต้องอ่าน weights ทั้งก้อนจาก memory การสร้างข้อความจึงถูกจำกัดด้วย "ความเร็วท่อส่งข้อมูล" (~273 GB/s บน DGX Spark) ไม่ใช่จำนวนการคูณเลข
2. **30 คนยิงพร้อมกัน = แบ่งบาริสต้าคนเดียว** — vLLM ใช้ **continuous batching** รวมทุกคำขอเข้าเป็นชุดเดียวเพื่อใช้ bandwidth ให้คุ้ม แต่ความเร็วต่อคน (per-user) ก็ยังต้องหารกัน
3. **2 ตัวเลขที่เราจะจดวันนี้:**
   - **TTFT (Time To First Token)** = หน่วงตั้งแต่กด Send จนตัวอักษรแรกโผล่ → "บาริสต้าเริ่มทำกี่วินาทีหลังรับออเดอร์"
   - **tok/s (tokens per second)** = พิมพ์ออกมาเร็วแค่ไหนหลังเริ่ม → "พิมพ์คำตอบไหลลื่นแค่ไหน"

> ความจุจริงของห้องนี้: worst case (30 คนยิงพร้อมกันเป๊ะ) ~12 tok/s/คน, จริง ๆ (แชตเป็น burst สลับกัน) ~30–46 tok/s/คน

## คันโยกเร่งความเร็ว (The Speed Levers)

นี่คือ 4 คันโยกที่ทีม serving หมุนเพื่อลด cost-per-token และเพิ่มความเร็ว ใน lab นี้เราจะ **เห็นผลของ 2 ตัวด้วยตาตัวเอง** (prefix caching + aggregate/per-user) ส่วนอีก 2 ตัวเป็น "ความรู้ที่ทีมเซ็ตไว้ให้แล้ว"

### 1. Quantization (NVFP4 / FP8) — "ย่อขนาดสูตร"

ลดความละเอียดของตัวเลขใน weights จาก FP16 → FP8 หรือ NVFP4 → โมเดลเล็กลง อ่านจาก memory ได้ไวขึ้น = **เร็วขึ้น + กิน memory น้อยลง** (เหลือที่ให้ KV cache มากขึ้น)

| รูปแบบ | ใช้กับอะไร | หมายเหตุสำคัญ |
|---|---|---|
| **FP8** | vLLM ใช้ได้ตรง ๆ (เช่น `--kv-cache-dtype fp8`) | DGX Spark รองรับ native |
| **NVFP4** | **ต้องใช้ TensorRT-LLM หรือ NIM เท่านั้น** | vLLM ทั่วไปยังไม่เสิร์ฟ NVFP4 → ถ้าจะใช้ ต้อง build engine ด้วย TensorRT-LLM หรือดึง NIM container |

> ⚠️ **จุดที่มือใหม่มักเข้าใจผิด:** "NVFP4 เปิดใน vLLM ได้เลย" — **ไม่ได้** NVFP4 ผูกกับ TensorRT-LLM / NIM วันนี้เราเสิร์ฟด้วย vLLM + FP8 เป็นหลัก แล้วพูดถึง NVFP4 เป็น "คันโยกขั้นถัดไป"

### 2. Prefix / Prompt Caching — "จำสูตรท่อนเปิดไว้" ⭐ (เราจะเห็นสด ๆ)

ถ้าหลายคำขอ **ขึ้นต้นเหมือนกัน** (เช่น system prompt เดียวกันทั้งห้อง) เซิร์ฟเวอร์ไม่ต้องประมวลผลท่อนเปิดซ้ำ → **cache hit** → TTFT ลดฮวบ นี่คือเหตุผลที่ห้องเราตั้ง **system prompt เดียวทั้งห้อง** เพื่อให้ทุกคนได้ประโยชน์ร่วมกัน

### 3. KV Cache — "โต๊ะเตรียมของ"

ระหว่างที่โมเดลสร้างข้อความ มันเก็บ "ความจำชั่วคราว" (Key/Value) ของทุก token ไว้ใน memory ยิ่งบทสนทนายาว/คนเยอะ → KV cache ยิ่งกิน memory vLLM ใช้ **PagedAttention** จัดสรร KV cache เป็นหน้า ๆ ให้ไม่เปลือง และตั้ง `--kv-cache-dtype fp8` เพื่อบีบ KV cache ให้เล็กลงครึ่งหนึ่ง

### 4. Right-sizing — "เลือกขนาดร้านให้พอดีงาน"

อย่าใช้โมเดลใหญ่เกินจำเป็น โมเดลคลาส MoE อย่าง Nemotron Nano (30B total / 3B active) เร็วเหมือนโมเดล 3B แต่ฉลาดกว่า → **ฟิตกับ DGX Spark + มือใหม่ที่สุด** และ **ห้ามโหลดโมเดลที่สองพร้อมกัน** (จะ OOM และแย่ง bandwidth)

> คำสั่ง serving ที่ห้องเราใช้ (instructor ตั้งไว้ให้แล้ว — ผู้เรียนไม่ต้องพิมพ์):
> ```bash
> vllm serve <model> \
>   --gpu-memory-utilization 0.85 \
>   --max-num-seqs 32 \
>   --max-model-len 4096 \
>   --kv-cache-dtype fp8
> ```
> - `--max-num-seqs 32` = รับได้สูงสุด 32 คำขอพร้อมกัน (พอดี ~30 คน)
> - `--max-model-len 4096` = จำกัด context ไม่ให้ KV cache บานจน OOM
> - `--gpu-memory-utilization 0.85` = กันที่ไว้ไม่ให้เต็ม 100% จนเครื่องสะดุด

## "cost per token = KPI" — ทำไมเรื่องนี้สำคัญ

ในโลก Agent โมเดลถูกเรียกซ้ำ ๆ เป็นลูป (REASON → ACT วนหลายรอบ) ไม่ใช่ถามครั้งเดียวจบ → **ทุก token ที่เปลืองคูณด้วยจำนวนรอบ** ทำให้ค่าใช้จ่ายบานเร็วมาก หนึ่งใน 4 หลักคิดของ "Agent Factory" คือ **cost per token = KPI ใหม่** (ตามที่ประกาศในงาน Blackwell ช่วยลด $/token ได้มาก) — แปลว่าเราต้องวัดและดูแลมันเหมือนวัดยอดขาย ไม่ใช่ปล่อยเลยตามเลย

---

## Context Rot — "window ใหญ่ != สมองใหญ่"

มือใหม่มักเชื่อว่า "context window ยิ่งใหญ่ยิ่งดี ยัดทุกอย่างเข้าไปเลย" — **นี่คือกับดัก**

- **ความจริง:** เมื่อ context ยาวขึ้นมาก ความแม่นยำของโมเดล **ตกลง** ไม่ใช่เพิ่มขึ้น (เรียกว่า **Context Rot**)
- **ตัวเลขอ้างอิง (ตามที่ประกาศในงาน):** f1 ตกจาก **73% ที่ 256k → 45% ที่ 1M tokens**
- **อุปมา:** ยัดเอกสาร 1,000 หน้าให้คนอ่านแล้วถามทันที กับให้สรุป 5 หน้าที่ตรงประเด็น — แบบหลัง "สมองทำงานดีกว่า" ทั้งที่อ่านน้อยกว่า

> ดังนั้นอย่ายัด context ใหญ่เกินจำเป็น เพราะ (1) มันช้าลง (KV cache บาน) (2) มัน **ฉลาดน้อยลง** ด้วย — เสียทั้งความเร็วและคุณภาพ

---

## มนตรา: `inline << file reference << var reference`

เวลาจะป้อนข้อมูลเข้า prompt มี 3 วิธี เรียงจาก "เปลือง token ที่สุด" ไป "ประหยัดที่สุด":

1. **inline** (แย่ที่สุด) = ก็อปเนื้อหาทั้งก้อนแปะลงใน prompt ตรง ๆ → กิน token ทุกครั้งที่เรียก
2. **file reference** (ดีกว่า) = อ้างถึงไฟล์ ให้ระบบดึงเฉพาะส่วนที่ต้องใช้
3. **var reference** (ดีที่สุด) = อ้างถึงตัวแปร/ผลลัพธ์ที่คำนวณไว้แล้ว ไม่ต้องส่งเนื้อหาดิบซ้ำ

> **จำง่าย ๆ:** `inline << file reference << var reference` (ยิ่งไปทางขวา ยิ่งประหยัด token) — อย่าแปะทุกอย่าง inline ถ้าอ้างอิงแทนได้

## ขั้นตอนปฏิบัติ (Step-by-step — no-code, คลิก/พิมพ์ในช่องแชตล้วน)

> ทุกคนทำผ่านหน้าเว็บ (Open WebUI / LM Studio shared endpoint) ที่ instructor เปิดให้ ไม่ต้องเขียนโค้ด
> instructor ฉาย **dashboard (Triton → Prometheus → Grafana)** บนจอใหญ่ตลอด lab

### Step 1 — Baseline: จด tok/s + TTFT (วัดของเดิมก่อน)

1. เปิดหน้าแชตที่ instructor แจก URL ให้
2. พิมพ์คำถามกลาง ๆ ที่ทุกคนใช้เหมือนกัน (instructor จะให้ prompt มาตรฐาน) เช่น:
   > `อธิบายว่า GPU ต่างจาก CPU อย่างไร แบบสั้น ๆ 3 ข้อ`
3. กด Send แล้ว **สังเกต 2 อย่าง:**
   - **TTFT** = นับในใจ "หนึ่ง...สอง..." ตั้งแต่กด Send จนตัวอักษรแรกโผล่ (ถ้าหน้าจอโชว์ตัวเลขให้ จดเลย)
   - **tok/s** = ดูว่าข้อความ "ไหล" เร็วแค่ไหน (หน้า WebUI/LM Studio มักโชว์ tokens/sec ท้ายคำตอบ)
4. **จดลงกระดาษ/แชต:** `รอบที่ 1 (baseline): TTFT ≈ ___ วินาที, tok/s ≈ ___`

### Step 2 — ส่ง prompt ซ้ำ → เห็น "prefix cache hit" (เร็วฮวบ) ⭐

1. **ส่ง prompt เดิมเป๊ะ ๆ อีกครั้ง** (อย่าแก้แม้แต่ตัวเดียว — เพราะ cache ทำงานเมื่อท่อนเปิดเหมือนเดิม)
2. สังเกต: **TTFT ควรลดลงชัดเจน** เพราะเซิร์ฟเวอร์ไม่ต้องประมวลผลท่อนเปิดซ้ำ → **prefix cache hit**
3. **จด:** `รอบที่ 2 (cache hit): TTFT ≈ ___ วินาที (เร็วขึ้นกว่ารอบแรก!)`
4. instructor ชี้ที่ dashboard: ตัว metric prefix cache hit rate / cache usage จะขยับขึ้น

> **บทเรียน:** นี่คือเหตุผลที่ห้องเราใช้ **system prompt เดียวกันทั้งห้อง** — ทุกคนแชร์ prefix เดียวกัน เซิร์ฟเวอร์เลยแคชท่อนเปิดให้ทั้งห้องได้ = ทุกคนได้ TTFT ดีขึ้น

### Step 3 — Aggregate vs Per-user: ดูภาพรวมเทียบรายคน

1. instructor นับ **3-2-1 ทุกคน Send พร้อมกัน** (ใช้ prompt เดียวกัน)
2. มองที่ dashboard บนจอใหญ่:
   - **Aggregate throughput (รวมทั้งเครื่อง)** = ตัวเลขนี้ **สูง** → "บาริสต้าทำกาแฟรวมได้เยอะมาก" (continuous batching ทำงาน)
   - **Per-user tok/s (รายคน)** = ตัวเลขนี้ **ต่ำลงกว่าตอนคนเดียว** → เพราะแบ่งบาริสต้าคนเดียวให้ 30 คน
3. **จด:** `ยิงพร้อมกัน 30 คน: per-user tok/s ≈ ___ (ต่ำกว่า baseline เพราะแชร์เครื่อง)`

> **บทเรียนสำคัญ (พูดดัง ๆ):** _"รวมแล้วเครื่องทำงานหนักและคุ้มมาก (aggregate สูง) แต่ความเร็วต่อคนลดลงตอนพร้อมกัน — นี่แหละเหตุผลที่เราต้องหมุนคันโยก Pillar 1 ตอนพร้อมกันช้าลง = บทเรียน ไม่ใช่บั๊ก"_
