# LAB 1 — LM Studio First-Win + Shared Endpoint + Concurrency Demo

> **Duration:** 10:45–11:20 (~35 นาที) · **Type:** Lab (no-code) · **Course:** DGX Spark — AI Nvidia 101 / Agent Enterprise
> **Hardware:** DGX Spark จริง 1 เครื่อง (GB10 Grace Blackwell, 128GB unified memory) · ผู้เรียน ~30 คน ต่อเข้าเครื่องเดียวกัน

---

## Objective (TH)

แล็บนี้ทำให้ผู้เรียน "ชนะตั้งแต่ก้าวแรก" (first-win) 3 เรื่อง โดยไม่ต้องเขียนโค้ดเลย:

- **(A)** เห็นกับตาว่า DGX Spark 1 เครื่องตรงหน้า = **ChatGPT ส่วนตัว** ที่รันอยู่ในห้องนี้เอง ผ่านโปรแกรม **LM Studio** ไม่ต้องต่อเน็ตออกไปหา cloud ของใคร
- **(B)** เข้าใจว่าทำไมทั้งห้อง 30 คน **ต่อเข้า "endpoint เดียวกัน"** (จุดเชื่อมต่อจุดเดียว) แทนที่จะลงโปรแกรมในเครื่องตัวเอง — และทำไม **LM Link ของ LM Studio ใช้ได้แค่ ~2 คน** จึงต้องเปลี่ยนไปใช้ **shared LAN endpoint** (vLLM/NIM + Open WebUI) สำหรับทั้งห้อง
- **(C)** ทดลอง **concurrency** ของจริง: นับ **"3-2-1 ทุกคน Send พร้อมกัน"** แล้วดู dashboard ว่าเกิดอะไรขึ้นกับ GPU และ memory bandwidth เมื่อ 30 คำขอวิ่งเข้าเครื่องเดียวพร้อมกัน

**ภาพใหญ่:** แล็บนี้คือฐานของทั้งวัน เพราะมันสอน "กติกาการแชร์เครื่องเดียว" ที่ทุกแล็บถัด ๆ ไปต้องใช้ และมันปูทาง **Pillar 1 (Optimize Tokens)** ด้วยการให้เห็นว่า "ช้าลงตอนยิงพร้อมกัน = บทเรียน ไม่ใช่บั๊ก"

# Part A — First-Win: LM Studio = ChatGPT ส่วนตัวบน Spark

## ภาพรวม (TH)

ก่อนจะพูดเรื่อง 30 คนแชร์เครื่อง ให้ผู้เรียนเห็น "ความมหัศจรรย์" ก่อน: เครื่องกล่องเล็ก ๆ บนโต๊ะตรงหน้า รันโมเดลภาษาขนาดใหญ่ได้เองโดยไม่ต้องพึ่งอินเทอร์เน็ต instructor จะฉาย **LM Studio** ขึ้นจอใหญ่แล้วพิมพ์คุยให้ดู

**Analogy (TH):** LM Studio เหมือน "เครื่องเล่นเพลงในบ้าน" — ไฟล์เพลง (โมเดล) อยู่ในเครื่องคุณเอง เปิดฟังได้แม้เน็ตดับ ต่างจาก ChatGPT ที่เหมือน "สตรีมมิ่งออนไลน์" ที่ต้องส่งทุกอย่างออกไปเครื่องคนอื่นบน cloud

## ขั้นตอน (instructor ทำหน้าจอ, ผู้เรียนดู)

1. instructor เปิดหน้าต่าง **LM Studio** ที่ฉายขึ้นจอ — ชี้ให้เห็นว่าโมเดลที่โหลดไว้คือ **Nemotron 3 Nano Omni** (โมเดลตัวอย่างจาก playbook, ขนาด ~65–70GB อยู่ใน unified memory 128GB)
2. instructor พิมพ์คำถามภาษาไทยง่าย ๆ ลงช่องแชต เช่น *"ช่วยแต่งกลอน 2 บรรทัดเรื่องแมวให้หน่อย"* แล้วกด **Send**
3. ชี้ให้ผู้เรียนดูว่า **ตอบกลับมาเลย** โดยไม่มีการต่อออกอินเทอร์เน็ต — โมเดลรันในเครื่องเองล้วน ๆ
   > ⚠️ **คำเตือนสำคัญ (TH):** ถ้าจะโชว์ "ตัดเน็ตแล้วยังตอบได้" ให้ทำ **เฉพาะกับเครื่อง/การเชื่อมต่อของ instructor เองในส่วน A (LM Studio)** เท่านั้น เช่น ปิด Wi-Fi ของเครื่อง instructor หรือถอดสายเน็ตขาออก (WAN/uplink) ของเครื่อง instructor — **ห้ามตัด LAN/Wi-Fi ของทั้งห้องเด็ดขาด** เพราะผู้เรียนทั้ง 30 คนพึ่ง **shared LAN endpoint** เดียวกันอยู่ ถ้าตัดเน็ตห้องจะทำให้ทุกคนหลุดจาก Open WebUI ทันที

## ผลที่คาดหวัง (Expected result)

- LM Studio พ่นคำตอบออกมาทีละคำ (streaming) บนจอ
- ผู้เรียน "อ๋อ" ว่ากล่องตรงหน้า = AI จริง ๆ ที่เป็นของเราเอง

## เกิดอะไรขึ้น (TH)

โมเดลภาษาทั้งก้อนถูกโหลดเข้าไปนั่งใน **memory ก้อนเดียว 128GB** ของ DGX Spark แล้ว เวลาเราถาม เครื่องก็คำนวณคำตอบจาก memory นั้นเลย ไม่ต้องส่งข้อความของเราออกไปไหน — **ข้อมูลไม่รั่วออกนอกห้อง** นี่คือหัวใจของแนวคิด "เอาโมเดลไปหา data" (data gravity) ที่จะพูดถึงทั้งวัน

> **ทำไม first-win นี้สำคัญ (TH):** ส่วน A ใช้เวลาสั้น ๆ แต่จำเป็น เพราะมันสร้าง "ความเชื่อ" ว่าเครื่องนี้ทำงานได้จริง ก่อนเราจะพาไปเจอความจริงที่ยากขึ้น (การแชร์ 30 คน) ในส่วน B และ C

# Part B — ทำไมต้อง Shared Endpoint (ไม่ใช่ LM Link)

## ภาพรวม (TH)

คำถามที่ผู้เรียนจะคิดทันทีคือ *"งั้นให้ทั้ง 30 คนเปิด LM Studio เหมือน instructor เลยสิ?"* — คำตอบคือ **ไม่ได้** และส่วนนี้อธิบายว่าทำไม พร้อมเฉลยวิธีที่ถูกต้อง

## ปัญหา: LM Link จำกัด ~2 คน

LM Studio มีฟีเจอร์แชร์ชื่อ **LM Link** แต่ ⚠️ **มันออกแบบมารับได้แค่ ~2 users** ถ้าจับ 30 คนยัดเข้าไป มันจะไม่ไหว นี่ไม่ใช่ของที่เอามาเสิร์ฟทั้งห้องได้

**Analogy (TH):** LM Link เหมือน "การแชร์จอกับเพื่อนข้าง ๆ 1–2 คน" ส่วนสิ่งที่เราต้องการสำหรับ 30 คนคือ "ร้านอาหารที่มีพ่อครัวกลาง 1 คน รับออเดอร์จากลูกค้าหลายโต๊ะพร้อมกัน"

## เฉลย: Shared LAN endpoint (vLLM/NIM) + Open WebUI

วิธีที่ถูกต้องสำหรับ 30 คนบนเครื่องเดียว:

- เครื่อง DGX Spark รัน **โมเดลเดียว** เปิดเป็น **endpoint กลาง** (OpenAI-compatible API) ด้วย **vLLM** (ตัวหลัก) หรือ **NIM** (ทางเลือกสำเร็จรูป)
- vLLM ใช้เทคนิค **PagedAttention + continuous batching** — รับหลายคำขอพร้อมกันแล้วจัดคิวรวมเป็นชุด (batch) ให้ GPU ทำทีเดียว เหมือนพ่อครัวรับหลายออเดอร์มาทำพร้อมกัน ไม่ใช่ทีละจาน
- ผู้เรียนทั้ง 30 คนเปิด **Open WebUI** (หน้าเว็บแชตหน้าตาเหมือน ChatGPT) ที่ชี้มายัง endpoint กลางนี้

**Analogy (TH):** endpoint กลาง = "เคาน์เตอร์รับออเดอร์จุดเดียว" · Open WebUI = "เมนู/หน้าจอสั่งของลูกค้าแต่ละคน" · vLLM continuous batching = "พ่อครัวที่ฉลาดพอจะทำหลายจานในกระทะเดียวรอบเดียว"

## ขั้นตอน (ผู้เรียนทำ — คลิก/พิมพ์เท่านั้น)

1. เปิดเบราว์เซอร์ ไปที่ URL บนการ์ด (เช่น `http://<spark-lan-ip>:3000`) — ถ้ายัง login ค้างจากช่วงลงทะเบียน ข้ามได้
2. ถ้าระบบให้ login: พิมพ์ **username/password บนการ์ด**
3. ดูมุมบน/กลางจอว่า **model ที่เลือกไว้** คือตัวที่ instructor ตั้งให้ (Nemotron Nano / 7–9B) — **ห้ามไปเปลี่ยนเป็นโมเดลอื่น**
4. พิมพ์ข้อความสั้น ๆ เช่น *"สวัสดี ทดสอบระบบ"* แล้วกด **Send** **(ยังไม่ต้องนับพร้อมกัน — แค่คนละจังหวะให้แน่ใจว่าต่อติด)**

## ผลที่คาดหวัง (Expected result)

- ทุกคนได้คำตอบกลับมาในหน้า Open WebUI ของตัวเอง
- ทุกคนคุยกับ **โมเดลตัวเดียวกัน** บนเครื่องเดียวกัน แม้จะเห็นเป็นหน้าแชตแยกของใครของมัน

## เกิดอะไรขึ้น (TH)

ทุกหน้าแชตของทุกคนวิ่งไปหา **จุดเดียวกัน** บน DGX Spark โมเดลถูกโหลดไว้ในเครื่อง **เพียงสำเนาเดียว** แล้ว vLLM ก็คอยจับคำขอจากทุกคนมารวมเป็นชุดให้ GPU ทำ — เพราะคอขวดจริงของ DGX Spark คือ **memory bandwidth (~273 GB/s)** ไม่ใช่ความจุ การรวมงานเป็น batch จึงเป็นวิธีใช้ bandwidth ให้คุ้มที่สุด

> **⚠️ กฎเหล็กของทั้งวัน (TH):** **ห้ามโหลดโมเดลที่สองพร้อมกัน** บนเครื่องนี้ เพราะจะแย่ง memory/bandwidth จนเครื่องล้ม (OOM) ทั้งวันเราใช้ "โมเดลเดียวที่แชร์ทั้งห้อง" เท่านั้น

# Part C — Concurrency Demo: "3-2-1 ทุกคน Send พร้อมกัน"

## ภาพรวม (TH)

นี่คือไฮไลต์ของแล็บ 1 และเราจงใจให้ "เครื่องช้าลง" เพื่อให้ผู้เรียนเห็นของจริง ไม่ใช่อุบัติเหตุ instructor จะฉาย **dashboard** ขึ้นจอใหญ่ค้างไว้ แล้วให้ทั้งห้องยิงคำขอพร้อมกัน

## เตรียมก่อนนับ (TH)

1. instructor ฉาย **Grafana dashboard** (GPU utilization, memory bandwidth, จำนวน requests กำลัง batch) ขึ้นจอใหญ่
2. ให้ทุกคน **พิมพ์ prompt เดียวกันที่เตรียมไว้** ลงช่องแชตของตัวเอง (เช่น *"อธิบายว่า DGX Spark คืออะไรใน 5 ประโยค"*) — **พิมพ์ไว้ก่อน แต่ยังไม่กด Send**
   - ใช้ prompt เดียวกันทั้งห้อง เพราะจะได้เห็นผลของ **prefix caching** ชัด ๆ (เก็บไว้ขยายใน LAB 3)

## ขั้นตอน (ผู้เรียนทำ)

1. instructor นับดัง ๆ: **"สาม… สอง… หนึ่ง… Send!"**
2. ทุกคนกด **Send พร้อมกัน**
3. ทุกคนมองทั้งหน้าจอตัวเอง **และ** dashboard บนจอใหญ่พร้อมกัน

## ผลที่คาดหวัง (Expected result)

- บน dashboard: **GPU utilization พุ่งขึ้น**, **memory bandwidth พุ่งเข้าใกล้เพดาน ~273 GB/s**, ตัวเลข **requests กำลัง batch** กระโดดขึ้นไปหลายสิบ
- หน้าแชตของทุกคน: คำตอบ **เริ่มไหลออกมาช้าลงกว่าตอนยิงคนเดียว** (ความเร็วต่อคนลดลง) แต่ **ทุกคนได้คำตอบครบ**
- ตัวเลขความเร็วโดยประมาณ (อ้างอิงการออกแบบคอร์ส): worst case ~**12 tok/s/คน**, จริงทั่วไป ~**30–46 tok/s/คน** เพราะแชตจริงมาเป็น burst ไม่ได้ยิงเป๊ะพร้อมกันตลอด

## เกิดอะไรขึ้น (TH)

ตอนยิงพร้อมกัน 30 คำขอ vLLM จับทั้งหมดมา **รวมเป็น batch เดียว** แล้วให้ GPU ทำรอบเดียว — GPU ก็เลย **busy เต็มที่** และ **memory bandwidth ถูกใช้จนเกือบเต็มเพดาน** เมื่อ bandwidth คือคอขวด ความเร็ว "ต่อคน" จึงลดลงเพราะทุกคนแชร์ท่อน้ำเดียวกัน

**สิ่งที่ instructor ต้องพูดดัง ๆ (สคริปต์):**
> *"เห็นไหมครับว่ามันช้าลงตอนเรายิงพร้อมกัน — **นี่คือบทเรียน ไม่ใช่บั๊ก** เพราะคอขวดของเครื่องนี้คือ memory bandwidth บ่ายนี้ใน Pillar 1 (Optimize Tokens) เราจะเรียนว่าจะทำให้มันเร็วขึ้นต่อคนได้ยังไง ด้วย quantization, caching และการจัดคิว"*

**Analogy (TH):** เหมือนทุกคนเปิดก๊อกน้ำจากท่อประปาเส้นเดียวกันพร้อมกัน — น้ำต่อคนเบาลง ไม่ใช่เพราะก๊อกพัง แต่เพราะท่อ (bandwidth) มีจำกัด

| login Open WebUI มีปัญหารายคน | ใช้ **การ์ดสำรอง** (เตรียมไว้ 5 ใบ) หรือให้จับคู่ดูจอเพื่อนชั่วคราว |
| เน็ต LAN/Wi-Fi รวน | ทุก asset (prompt ตัวอย่าง, dashboard) ใช้งานในวง LAN อยู่แล้ว ไม่พึ่งเน็ตนอก; ถ้าจำเป็นจริง ๆ ฉายคลิป dry-run ที่อัดไว้ |

> **หลักคิด fallback (TH):** เป้าหมายคือ "ทุกคนได้เห็น concurrency อย่างน้อย 1 ครั้ง" — ถ้าทั้งห้องพร้อมกันไม่ไหว ให้ลดเป็นเวฟ; ถ้าเวฟยังไม่ไหว ให้ฉายของ instructor; ห้ามปล่อยให้แล็บจบโดยไม่มี visible win

## Instructor Setup (กล่องคำสั่งฝั่งผู้สอน — ไม่ใช่สิ่งที่ผู้เรียนพิมพ์)

> ⚠️ คำสั่งทั้งหมดนี้ instructor รัน/ตั้งค่า **ก่อนวันงาน (dry-run)** บน DGX Spark เท่านั้น ผู้เรียนไม่แตะ terminal เลย
> 🔴 **ห้ามรัน `curl … | bash` สดต่อหน้าผู้เรียน** — ติดตั้งและทดสอบทุกอย่างใน dry-run ก่อน

### A) LM Studio (first-win demo) — headless `llmster` + GUI

```bash
# ติดตั้ง LM Studio headless (llmster)
curl -fsSL https://lmstudio.ai/install.sh | bash

# ดึงโมเดลตัวอย่างจาก playbook (Nemotron 3 Nano Omni, ~65–70GB)
lms get nvidia/nemotron-3-nano-omni

# โหลดโมเดลเข้า memory
lms load nvidia/nemotron-3-nano-omni

# เปิด server (bind LAN, port 1234) — สำหรับ demo Part A
lms server start --bind 0.0.0.0 --port 1234
```

GUI (สำหรับโชว์บนจอ): **Developer → Serve on Local Network**

> หมายเหตุ: LM Studio ใช้สำหรับ **Part A (first-win) เท่านั้น**. ฟีเจอร์ **LM Link จำกัด ~2 users** ใช้กับ 30 คนไม่ได้ → ไปใช้ vLLM/NIM ใน Part B/C.

### B) Serving endpoint สำหรับทั้งห้อง (workhorse = vLLM)

โมเดลที่แชร์ทั้งห้อง = **Nemotron Nano (30B-A3B MoE)** หรือคลาส 7–9B (FP8/NVFP4):

```bash
# vLLM = ตัวหลัก: OpenAI-compatible, PagedAttention, continuous batching
vllm serve <model> \
  --gpu-memory-utilization 0.85 \
  --max-num-seqs 32 \
  --max-model-len 4096 \
  --kv-cache-dtype fp8
```

ทางเลือก: **NIM** (endpoint สำเร็จรูป, OpenAI endpoint `/v1/chat/completions` พอร์ต 8000):

```bash
docker run --gpus all --shm-size=16GB \
  -e NGC_API_KEY=$NGC_API_KEY \
  -v "$LOCAL_NIM_CACHE:/opt/nim/.cache" \
  -p 8000:8000 \
  nvcr.io/nim/meta/llama-3.1-8b-instruct-dgx-spark:latest
```

> ⚠️ **NVFP4** ใช้ได้เฉพาะกับ **TensorRT-LLM หรือ NIM** เท่านั้น (vLLM ใช้ FP8). อย่าโหลดโมเดลที่สองพร้อมกัน = OOM.

### C) Front-end + dashboard

- **Open WebUI**: ชี้ไปยัง endpoint vLLM/NIM (OpenAI-compatible) ให้ผู้เรียน 30 คน login ผ่าน browser; bind ในวง LAN เท่านั้น
- **Dashboard**: **Triton → Prometheus → Grafana** สำหรับโชว์ GPU utilization + memory bandwidth + จำนวน requests ที่กำลัง batch ตอน "3-2-1 Send พร้อมกัน"

## สิ่งที่ instructor ต้องเช็ค/เติมเอง (TH)

- **IP/URL จริง** ของ Open WebUI (`<spark-lan-ip>`) และ port → เขียนลงการ์ด + กระดานก่อนเริ่ม
- **ชื่อโมเดลจริง** ที่จะเสิร์ฟ (Nemotron Nano หรือคลาส 7–9B ตัวที่เลือก) → ใส่แทน `<model>` ในคำสั่ง vLLM
- **ตัวเลข tok/s จริง** ต้องวัดใน **dry-run 30 บัญชียิงพร้อมกัน** ก่อนวันงาน (12 / 30–46 tok/s/คน เป็นค่าออกแบบ ยังไม่ยืนยัน)
- ค่า `--max-num-seqs` / `--max-model-len` / `--gpu-memory-utilization` ปรับจากผล dry-run จริง
- เตรียม **prompt กลางทั้งห้อง** (สำหรับ Part C) + **คลิป concurrency สำรอง** จาก dry-run เผื่อ demo สดล่ม
- ยืนยัน **dashboard** ฉายขึ้นจอ + ตัวเลขขยับจริงตอนยิงพร้อมกัน
