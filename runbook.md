## PRE-FLIGHT เช้า (08:30–10:00) — เช็คก่อนเปิดประตู

> ทำตามลำดับนี้ทุกข้อ **อย่าข้าม** — เป้าหมาย: เมื่อผู้เรียนคนแรกเดินเข้ามา ระบบ "อุ่นเครื่องพร้อมยิง" แล้ว
> รายละเอียดคำสั่งเต็ม + ค่าตัวเลขที่ถูกต้องอยู่ใน `infra-setup-checklist.md` — ที่นี่คือ **ลำดับเช้าวันงาน** ที่ต้องไล่จริง

### 08:30 — เปิดเครื่อง + ยืนยัน serving stack ขึ้น
- [ ] เปิด DGX Spark, ต่อจอใหญ่/โปรเจกเตอร์, ยืนยันเห็นเดสก์ท็อป
- [ ] เปิด terminal ตรวจว่า **vLLM (ตัวหลัก) เสิร์ฟโมเดลเดียวทั้งห้องอยู่** — โมเดลแชร์ = **Nemotron Nano (30B-A3B MoE)** หรือคลาส 7–9B (FP8/NVFP4)
  ```
  vllm serve <model> --gpu-memory-utilization 0.85 --max-num-seqs 32 --max-model-len 4096-8192 --kv-cache-dtype fp8
  ```
- [ ] (ถ้าใช้ NIM แทน) ยืนยัน container ขึ้นพอร์ต 8000, endpoint `/v1/chat/completions` ตอบได้
- [ ] ยิง prompt ทดสอบ 1 อันจากเครื่อง instructor → ต้องได้คำตอบกลับ

### 08:45 — RE-WARM โมเดล (สำคัญที่สุด)
> โมเดลที่เพิ่งโหลดยัง "เย็น" — request แรกจะช้าผิดปกติ (TTFT สูง) เพราะยังไม่มี cache/JIT warm
- [ ] ยิง **prompt ซ้อม 5–10 อัน** ผ่าน shared endpoint ให้โมเดลอุ่น (ใช้ system prompt เดียวกับที่จะใช้ในห้อง เพื่อให้ **prefix cache** ติดไว้ล่วงหน้า)
- [ ] ดู dashboard: tok/s ควรนิ่งขึ้น, TTFT ลดลงหลัง request ที่ 2–3
- [ ] **ยืนยัน: มีโมเดลตัวเดียวบนเครื่อง** — ไม่มีโมเดลที่สองค้างจาก dry-run เมื่อวาน (ถ้ามี = OOM แน่นอน → kill ทิ้งก่อน)

### 09:00 — เช็ค 30 KERNELS / 30 บัญชี
- [ ] เปิด JupyterHub admin → ยืนยัน **30 บัญชีผู้เรียน pre-provision ครบ** + Open WebUI ชี้มาที่ endpoint เดียว
- [ ] สั่ง **spawn/warm ทั้ง 30 kernels ล่วงหน้า** (อย่าให้ผู้เรียนต้องรอ kernel boot ตอน 10:45) → ยืนยันทุกบัญชีมี kernel เขียว
- [ ] ทดสอบ login จริง 2–3 บัญชีสุ่ม (รวมการ์ดสำรอง 1 ใบ) → ต้องเข้าได้ + รัน cell "hello" ได้
- [ ] ยืนยัน **การ์ด URL + user/pass ครบ 30 ใบ + สำรอง 5 ใบ** วางที่โต๊ะลงทะเบียน

### 09:20 — โหลด NOTEBOOK ทุก LAB ค้างไว้
- [ ] เปิด **notebook ของ LAB 1–5 ค้างไว้** บนเครื่อง instructor (template เดียวกับที่ผู้เรียนจะเปิด) → จะได้สลับสอนได้ทันทีไม่ต้องหา
- [ ] ยืนยัน notebook ทุก lab รัน cell แรกได้ (ไม่ค้าง import / ไม่ฟ้อง endpoint ไม่เจอ)
- [ ] เปิด **asset offline** เตรียมไว้: คลิป G1 humanoid (Jetson Thor, MuJoCo-Warp), GTC recap board, สไลด์ AI-Q / OpenShell → ยืนยันเล่นได้ **โดยถอดเน็ต** (กันเน็ตล่มตอนฉาย)

### 09:35 — ฉาย DASHBOARD ขึ้นจอที่สอง
- [ ] เปิด **Triton dashboard (Prometheus → Grafana)** ขึ้นจอที่สอง/โปรเจกเตอร์ตัวที่สอง → ตั้งให้เห็น: GPU util %, memory ใช้ไป/เหลือ, tok/s รวม, batch size, จำนวน active requests
- [ ] ทดสอบ: ยิง 3–4 prompt → เห็นกราฟขยับจริง (ยืนยัน dashboard ไม่ค้างที่ค่าศูนย์)
- [ ] วางจอ dashboard ให้ผู้เรียน "เห็นได้ทั้งห้อง" — นี่คือ **พระเอกของวัน** (เราจะชี้กลับมาตลอด)

### 09:50 — Final go/no-go

- [ ] หายใจเข้าลึก ๆ เปิดประตู

> **PRE-FLIGHT mantra:** *re-warm โมเดล → 30 kernels เขียว → notebook ค้างพร้อม → dashboard ขึ้นจอ.* ครบ 4 อย่างนี้ค่อยเปิดประตู

## ประโยคสคริปต์หลักประจำวัน (ท่องให้ขึ้นใจ)

ประโยคนี้คือ **หัวใจการสอน Pillar 1** และเป็นตัวกันไม่ให้ contention กลายเป็น "บั๊กที่ทำให้ผู้เรียนเสียกำลังใจ" ใช้ทุกครั้งที่ทั้งห้องยิงพร้อมกันแล้วช้าลง:

> ### 🗣️ **"ช้าตอนพร้อมกัน = บทเรียน ไม่ใช่บั๊ก"**
>
> *"เห็นไหมครับ ตอนเราคนเดียวยิง มันเร็วมาก แต่พอ 30 คนกด Send พร้อมกัน มันช้าลง — **นี่ไม่ใช่เครื่องเสีย ไม่ใช่บั๊ก นี่คือบทเรียนข้อแรกของเรา**. เครื่องเดียวมีพลังจำกัด เหมือนร้านกาแฟร้านเดียวที่ลูกค้า 30 คนสั่งพร้อมกัน — บาริสต้าคนเดิมต้องทำทีละแก้ว. คอขวดของเราคือ **memory bandwidth** ไม่ใช่จำนวนเครื่อง. ตอนบ่ายเราจะเรียนวิธี 'ทำให้ร้านนี้เสิร์ฟเร็วขึ้น' โดยไม่ต้องซื้อเครื่องเพิ่ม — นั่นคือ Pillar 1: Optimize Tokens"*

ใช้ซ้ำได้ทุกบล็อกที่มี concurrency (LAB 1, LAB 3, LAB 4) — เปลี่ยนเป็น **"จุดขาย" ไม่ใช่ "คำขอโทษ"**

## บล็อก 2 — Welcome + "กล่องนี้คืออะไร" (DGX Spark) (10:25–10:45, 20 นาที)

**DO**
- ฉาย 1 สไลด์ภาพ scale: **small → desktop → datacenter** (DGX Spark = "datacenter ขนาดวางบนโต๊ะ")
- ฉาย dashboard เครื่องจริงคู่กัน → ชี้ว่า "นี่คือเครื่องที่ทุกคนเพิ่งทักทายไป"
- เปิดภาพแกนกลางของวัน: **"Agent = Model + Harness"** (loop CONTEXT → OBSERVE → REASON → ACT) — บอกว่าจะกลับมาที่ภาพนี้ทุกช่วง

**SAY**
- *"กล่องนี้ชื่อ DGX Spark ข้างในคือชิป GB10 — รวม CPU แบบ Grace กับ GPU แบบ Blackwell ไว้ด้วยกัน มี memory ก้อนเดียว 128GB ที่ทั้ง CPU และ GPU ใช้ร่วมกัน (unified memory). เปรียบเหมือน **ครัวที่เชฟกับผู้ช่วยใช้เคาน์เตอร์เดียวกัน** ส่งของให้กันได้เร็วมาก"*
- *"แต่จุดสำคัญที่ต้องจำทั้งวันคือ: คอขวดของเครื่องนี้ไม่ใช่ 'พื้นที่' (128GB เยอะมาก) แต่คือ **ความเร็วในการขนข้อมูลเข้าออก memory** (~273 GB/s). เก็บคำนี้ไว้นะครับ เดี๋ยวบ่ายมันจะอธิบายว่าทำไม 30 คนยิงพร้อมกันถึงช้าลง"*
- *"และทั้งวันนี้เราจะสร้างสิ่งที่เรียกว่า **Agent**. สูตรง่าย ๆ คือ **Agent = Model + Harness** — Model คือสมอง, Harness คือ 'ร่างกายกับกฎ' ที่ให้สมองทำงานวนเป็นลูป: รับบริบท → สังเกต → คิด → ลงมือ. จำภาพนี้ไว้ครับ"*

## บล็อก 8 — LAB 3: Pillar 1 Optimize Tokens (13:10–13:50, 40 นาที)

- ฉาย dashboard เทียบ **aggregate (รวมทั้งห้อง) vs per-user** → เห็นว่ารวมแล้ว throughput สูง แต่ต่อคนลดลงตอนยิงพร้อมกัน

- เกริ่นแนวคิดลึก: **"cost per token = KPI ใหม่"**, **Context Rot** ("window ใหญ่ ≠ สมองใหญ่" — f1 73%@256k → 45%@1M), มนตรา **"inline << file reference << var reference"**

**SAY**

- ย้ำสคริปต์ "ช้าตอนพร้อมกัน = บทเรียน" + ต่อ: *"วิธีแก้ไม่ใช่ซื้อเครื่องเพิ่ม แต่คือใช้คันโยกพวกนี้ — ลดขนาดข้อมูลต่อ token (quantization), จำของซ้ำ (caching), เลือกโมเดลให้พอดีงาน (right-sizing)"*
- KPI: *"ในโลก agent จริง เราไม่ได้วัดแค่ 'ตอบถูกไหม' แต่วัด 'ตอบถูกโดยจ่ายกี่ token' — **cost per token คือ KPI ใหม่**"*
- Context Rot: *"และยัดข้อมูลเข้าไปเยอะ ๆ ไม่ได้แปลว่าฉลาดขึ้นนะครับ — window ใหญ่ ≠ สมองใหญ่ ใส่เยอะเกินมันกลับเบลอ"*

# 🚨 FAILURE-RECOVERY PLAYBOOK

> เปิดหน้านี้ค้างไว้ในมือถือ/แท็บเล็ตของ Lead + Ops ตลอดวัน
> **หลักคิดรวม:** อย่าตื่นตระหนกต่อหน้าผู้เรียน — ทุกปัญหาแปลงเป็น "บทเรียน" ได้. Ops แก้เงียบ ๆ, Lead พูดกลบด้วยเนื้อหา

---

## 1. DGX Spark OVERLOAD (เครื่องช้าผิดปกติ / tok/s ตก / batch ล้น)

**อาการ:** dashboard เห็น GPU util เต็ม 100% ค้างนาน, tok/s/คนต่ำกว่าปกติมาก, request เข้าคิวยาว, ผู้เรียนบ่น "ค้าง"

**Ops ทำ (เงียบ ๆ):**
1. **ลด batch / จัดเป็นเวฟ** — ลด `--max-num-seqs` (เช่น 32 → 16) หรือให้ผู้เรียนยิงเป็น 2 เวฟ (ครึ่งห้องก่อน)
2. **cap context + max_tokens** ให้ต่ำลงชั่วคราว (ลดภาระต่อ request)
3. ยืนยันไม่มีโมเดล/process แปลกปลอมแย่ง bandwidth

**Lead พูด (แปลงเป็นบทเรียน):**
> *"นี่แหละครับสิ่งที่เราพูดไว้ — ช้าตอนพร้อมกัน = บทเรียน. ตอนนี้เราจะ 'จัดคิว' ให้ดูเลยว่าทำไมการแบ่งเวฟช่วยได้"* → ใช้โอกาสนี้สอน Pillar 1 สด

**ถ้ายังไม่ดีขึ้น:** ลด concurrency demo เหลือ instructor ยิงโชว์คนเดียว + ฉาย dashboard

---

## 2. OOM (Out of Memory — มักเกิดจากโหลดโมเดลที่สอง)

**อาการ:** memory บน dashboard พุ่งชนเพดาน, serving crash / ไม่ตอบ, log ฟ้อง out of memory

**สาเหตุอันดับ 1:** มีคน/process **โหลดโมเดลที่สองพร้อมกับโมเดลหลัก** (เช่น เผลอ load ใน LM Studio, NemoGuard NIM 8B 16–20GB, Ollama ใหญ่เกิน)

**Ops ทำทันที:**
1. **หา + kill โมเดล/process ตัวที่สอง** ที่เพิ่งโหลด → คืน memory
2. **รีสตาร์ท serving โมเดลเดียว** (โมเดลหลัก vLLM/NIM) ให้กลับมาเสิร์ฟ
3. **re-warm** ด้วย 5 prompt ก่อนปล่อยทั้งห้องกลับมายิง
4. ยืนยัน dashboard: 1 โมเดลเดียว, memory เหลือพอสำหรับ KV cache

**Lead พูด (กลบเวลา):**
> *"ระหว่างที่ทีมงานจัดเครื่องสักครู่ ผมขออธิบายว่าทำไมเราถึง 'ห้ามโหลดโมเดลที่สอง' บนเครื่องเดียว — 128GB ฟังดูเยอะ แต่ KV cache (ความจำของบทสนทนา) กินที่เยอะมาก..."* → เปลี่ยนเป็นบทเรียน unified memory

**กันล่วงหน้า:** Floor Helper เฝ้าไม่ให้ผู้เรียนกด load โมเดลเองใน LM Studio; toggle `/think` `/no_think` ใช้โมเดลเดิมเสมอ

---

## 3. เน็ตล่ม (LAN/Wi-Fi/internet ขัดข้อง)

**แยก 2 กรณี:**

**กรณี A — LAN ในห้องล่ม (ผู้เรียนต่อ endpoint ไม่ได้):**
- Floor Helper เช็ค AP/สวิตช์ก่อน (เสียบหลวม? reboot?)
- ถ้า LAN ใช้ไม่ได้ชั่วคราว → Lead **โชว์จากเครื่อง instructor บนจอ** ให้ดูแทน, ผู้เรียนดูตาม, แล้วค่อยให้ลงมือเมื่อ LAN กลับมา

**กรณี B — internet ขาออกล่ม (แต่ LAN ในห้องยังดี):**
- **ส่วนใหญ่ไม่กระทบ!** โมเดลรันบนเครื่องในห้อง (offline by design), shared endpoint อยู่ใน LAN
- สิ่งที่กระทบ = asset ที่ต้องดึงสด → เราเซฟ **offline ไว้หมดแล้ว** (คลิปหุ่นยนต์, GTC recap, สไลด์)
- LAB 5 demo `curl` ออกเน็ต: ถ้า internet ล่มจริง → `curl` ก็จะ fail อยู่แล้ว แต่ **demo เราต้องการให้ OpenShell เป็นคนบล็อก (403 from proxy) ไม่ใช่เน็ตล่ม** → ถ้าเน็ตล่ม ให้ Lead อธิบายด้วย screenshot/clip ที่เตรียมไว้ และพูดชัดว่า "ปกติ OpenShell จะเป็นคนบล็อกตรงนี้"

**Lead พูด:** *"ข่าวดีคือ AI ของเรารันในห้องนี้ ไม่ต้องพึ่งเน็ตข้างนอก — นี่คือข้อดีของการมีเครื่องเป็นของตัวเอง ข้อมูลไม่รั่ว เน็ตล่มก็ยังทำงานได้"* (เปลี่ยนเป็นจุดขาย data gravity / security-by-default)

## 4. DEMO สด FAIL (คำสั่งไม่รัน / notebook error / ผลไม่ออกตามคาด)

**กฎทอง:** **อย่าซ่อมสดต่อหน้าคนเกิน 60 วินาที** — ถ้าเกิน ตัดไป fallback ทันที

**ลำดับ recovery:**
1. **ลองรันซ้ำ 1 ครั้ง** (บางทีแค่ request แรกเย็น/timeout)
2. ถ้ายังไม่ได้ → **สลับไปใช้ notebook ที่ instructor โหลดค้างไว้** (เครื่อง instructor) แทนของผู้เรียน
3. ถ้ายังไม่ได้ → **เปิดคลิป/screenshot สำรอง** ที่เตรียมไว้ของ demo นั้น แล้วเล่าจากภาพ
4. โดยเฉพาะ **LAB 4 full multi-agent loop** (เสี่ยงสุด): มี **cloud fallback** + คลิปสำรอง — สลับได้ทันที

**Lead พูด (กลบให้เนียน):**
> *"เครื่อง demo สดบางทีก็ดื้อแบบนี้แหละครับ — ซึ่งจริง ๆ มันสอนเราอย่างนึง: ในงานจริง agent ต้องมี fallback เสมอ. ผมมีคลิปที่อัดไว้ตอนซ้อม มาดูผลกัน"*

**กันล่วงหน้า:** pre-flight โหลด notebook ทุก lab ค้างไว้ + เตรียมคลิป/screenshot สำรองทุก demo เสี่ยง
